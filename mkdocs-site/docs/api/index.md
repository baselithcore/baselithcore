## OpenAPI Specification

The framework automatically exposes a complete OpenAPI 3.0 specification, dynamically generated by FastAPI, which includes:

- **Request schema**: All required parameters, body, and headers
- **Response schema**: Return data structure for each endpoint
- **Authentication**: Supported methods (API Key, JWT)
- **Examples**: Sample request/response for each endpoint
- **Error codes**: Complete documentation of possible errors

### Interactive Documentation Access

Three different interfaces are available to explore and test the APIs:

#### Swagger UI

**URL**: `http://localhost:8000/docs`

Swagger UI is the most complete **interactive** interface. It allows you to:

- **Explore** all endpoints organized by tag (Core, Admin, Plugin)
- **Test** APIs directly from the browser without Postman or curl
- **See** complete JSON schemas with real-time validation
- **Authenticate** by entering API Key or JWT token once ("Authorize" button)

**When to use it**:

- During development to quickly test new endpoints
- To explore APIs of a newly installed plugin
- For debugging integration issues

!!! tip "Quick Test"
    1. Open `http://localhost:8000/docs`
    2. Click "Authorize" in the top right
    3. Enter your API Key
    4. Expand an endpoint (e.g. `POST /api/chat`)
    5. Click "Try it out"
    6. Modify the example JSON
    7. Click "Execute" and see the live response

#### ReDoc

**URL**: `http://localhost:8000/redoc`

ReDoc provides highly readable **static documentation**. Optimized for:

- **Reading** and understanding API architecture
- **Quick search** between endpoints (search box)
- **Navigation** via sidebar index
- **Export** or sharing with the team

**When to use it**:

- When you want a complete overview of available APIs
- For documentation to share with frontend developers
- To generate API documentation screenshots

#### OpenAPI JSON

**URL**: `http://localhost:8000/openapi.json`

Raw OpenAPI schema in JSON format. Useful for:

- **Import** into external tools (Postman, Insomnia, Paw)
- **Generation** of automatic client SDKs
- **CI/CD** spec validation
- **Versioning** API documentation

**When to use it**:

- Import to Postman: Download JSON → Import to Postman Collections
- Client generation: Use `openapi-generator` to create TypeScript/Python/etc clients
- Automatic testing: Validate responses against the schema

### Custom Documentation Generation

You can generate additional documentation using the CLI:

```bash
# Generate OpenAPI YAML and Postman Collection files
baselith docs generate
```

This creates:

- `mkdocs-site/docs/api/specs/openapi.yaml` - Spec in YAML format (more human-readable)
- `mkdocs-site/docs/api/specs/openapi.json` - Spec in JSON format
- `mkdocs-site/docs/api/specs/postman_collection.json` - Collection importable to Postman

### Plugin Documentation

Each plugin automatically registers its endpoints in the OpenAPI documentation. When you load a new plugin, its endpoints immediately appear in Swagger UI under a tag with the plugin name.

**Example**: If you install the `weather-agent` plugin, you'll see a new "Weather" tag with all `/api/weather-agent/*` endpoints.

### Security in Documentation

!!! warning "Production"
    In production environments, **disable** Swagger UI and ReDoc to avoid information disclosure:

    ```env
    ENABLE_API_DOCS=false  # Disables /docs and /redoc
    ```
    
    The OpenAPI JSON schema will remain available only for authenticated users.

### Best Practices

!!! tip "Versioning"
    Commit `mkdocs-site/docs/api/specs/openapi.yaml` to the repository to track API changes over time:

    ```bash
    git add mkdocs-site/docs/api/specs/openapi.yaml
    git commit -m "docs: Update API spec with new weather endpoints"
    ```

!!! tip "Client Generation"
    Generate automatic clients for frontend:

    ```bash
    # TypeScript/JavaScript client
    openapi-generator-cli generate \
      -i http://localhost:8000/openapi.json \
      -g typescript-axios \
      -o frontend/src/api-client
    ```
